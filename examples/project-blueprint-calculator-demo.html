A modern calculator that respects math

A fast, keyboard-first web calculator that evaluates whole expressions with correct operator precedence, keeps a running history tape, and stays accurate where naive calculators embarrass themselves. Clean by default, scientific when you need it.

web app expression evaluation scientific toggle keyboard-first offline / PWA
01

Problem & motivation

The calculators people reach for are split into two bad extremes. The default OS/phone calculator is often too basic — it applies operators left-to-right, so 3 + 4 × 2 gives the wrong answer, and there's no history to check your work. The "pro" apps swing the other way: dense scientific grids that intimidate the 90% who just want 12 × 7. Neither is pleasant, and both quietly lie about floating point (0.1 + 0.2). This project is a calculator that is correct (real precedence, sane rounding), fast (keyboard-first, instant), and calm (basic by default, scientific one tap away).

02

Goals & non-goals

Goals
  • Evaluate full expressions with correct precedence & parentheses.
  • Accurate to ~12 significant figures; no 0.1+0.2 artifacts on screen.
  • Full keyboard support; every action reachable without the mouse.
  • A persistent history tape you can reuse and edit.
  • Progressive: clean basic keypad, scientific mode behind a toggle.
  • Themeable (light/dark) and works offline.
Non-goals
  • Not a graphing or symbolic (CAS) calculator in v1.
  • No unit / currency conversion (defer to v2).
  • No cloud sync or accounts — client-only.
  • No programmer mode (hex/binary/bitwise) yet.
  • Not a spreadsheet; one expression at a time.
03

Target users

  • Everyday user — quick sums, tips, percentages; wants it to just be right and fast.
  • Student — needs precedence, parentheses, powers/roots, and a visible history to check steps.
  • Developer / power user — keyboard-only flow, reuses previous results, expects no float surprises.
04

User stories

  • As a user, I type 3+4*2 and press Enter, so that I get 11 — the mathematically correct answer.
  • As a user, I see my last calculations in a tape, so that I can tap one to reuse its result.
  • As a student, I toggle scientific mode, so that I can use √, xʸ, and sin/cos/log.
  • As a keyboard user, I never touch the mouse — digits, operators, Enter, Esc, and Backspace all work.
  • As a returning user, my history is still there after I reload the page.

06

Non-functional requirements

  • Performance — input-to-display latency < 16 ms (one frame); evaluation of a normal expression is instant.
  • Accuracy — results correct to 12 significant figures; display never shows float artifacts.
  • Accessibility — full keyboard operation, ARIA roles on the keypad, visible focus, screen-reader-announced results.
  • Offline — a service worker caches the app shell; fully usable with no network.
  • Footprint — no framework; ships as a small static bundle.

08

Key user flows

Evaluate an expression (the core flow)

Type 3+4×2 → each keypress updates the display → press =/Enter → tokenize → shunting-yard to RPN → evaluate → round for display → show 11 and push 3+4×2 = 11 to the history tape.

Reuse from history

Tap a tape entry → its result seeds the next expression (or its full expression is loaded for editing).

09

Milestones

  • M0 — tokenizer + shunting-yard evaluator + unit tests (the engine, headless).
  • M1 — keypad UI + display + the input state machine; basic operators.
  • M2 — history tape + localStorage; keyboard support; error handling.
  • M3 — scientific toggle, memory, theming.
  • M4 — PWA / offline, accessibility pass, polish.
10

Success metrics

  • 100% of a precedence/associativity/parentheses test suite passes.
  • Zero float-artifact displays across a fuzz test of random expressions.
  • Every function reachable by keyboard; audited with a screen reader.
  • App shell loads & runs with the network disabled.
11

Risks & open questions

  • Assumption: a client-only web app in vanilla JS; numbers are IEEE-754 doubles with display rounding (see the Decisions tab for the trade-off vs a decimal library).
  • Risk: display rounding can hide genuine tiny results — mitigate by rounding to significant figures, not fixed decimals, and keeping full precision internally.
  • Risk: percent (%) is context-dependent (200+10% vs 10% of what?) — define its semantics explicitly and test them.
  • Open: should tapping history reuse the result or reload the expression for editing? (Leaning: result on tap, expression on long-press.)
